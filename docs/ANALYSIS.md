# Analysis — issue #8: architecture review with step-by-step feedback

This is the written answer to issue #8. The graphical report it keeps pointing at
is in [`docs/report/REPORT.md`](report/REPORT.md) (eight figures, regenerated by
`experiments/report.py`); the figures are embedded inline below where they carry
the argument.

Everything here obeys the same provenance rule as the rest of `docs/`: a number is
**MEASURED** (a training run or a cited primary source), **DERIVED** (computed from
measured inputs), or **PROJECTED** (a scaling-law extrapolation). The three are
never blurred, and where a run did not happen the report says so rather than
guessing.

## 0. TL;DR

- **What PR #7 is:** it untied the weight-shared loop. `quark_3m` (1 unique layer
  run 12 times, weights tied) became `quark_22m` (12 unique layers, run once) —
  same compute graph, ~7.6× the stored parameters. MEASURED, that single change
  cut WikiText-103 word perplexity **108 → 74.965** and lifted BLiMP **58.6 →
  61.76**, at **+0 FLOPs and +0.30 GB VRAM**. It is the strongest result in the
  project so far, and it reorganised the whole thesis (see §1).
- **The decided path:** the limiting factor is **not** parameters and **not** the
  architecture — it is the **token budget under a 16 GB VRAM ceiling**. `quark_22m`
  has seen ~0.33× a Chinchilla-optimal number of tokens. So the next runs spend on
  *more tokens per parameter* (4 epochs + dropout + weight-decay sweep) and on
  *freeing VRAM to afford them* (flash attention, larger batch), before anything
  exotic. This is the gated order in Fig 6 / NEXT.md §13.
- **The backend question (wgpu/vulkan/rocm):** answering it requires the GPU, and
  the machine I run on has none. The first attempt also never reached the runner
  for a second reason the maintainer spotted: the workflow targeted
  `runs-on: [self-hosted, gpu]`, but the runner is matched by its **label**
  (`self-hosted`) — `gpu` is only its *name*, not a label — so no runner ever
  matched the job. That is now fixed (`runs-on: self-hosted`), so a dispatch from
  the upstream repo reaches the runner and runs the harness end to end. Until that
  run completes, the report renders honest placeholders for the two GPU-only panels
  (Fig 7 backends, Fig 8 sweep). See §5 and §7.
- **One thing I did land in code:** NEXT.md §13's *step 1* — the warmup-unit fix —
  because it is pure CPU logic and the sweep is meaningless without it. `TrainConfig`
  now has a run-relative `warmup_ratio` (§6).

## 1. What PR #7 is, and the essence of the project

**The project.** quark is a small causal language model in Rust on `burn` 0.21,
aiming at the issue's goal: *a universal, fast, small GPT-2-level transformer with
good reasoning that a person can train in reasonable time on 16 GB of VRAM.* The
reference model is tiny — vocab 8192, `d_model` 384, 6 heads (2 KV via GQA), RoPE,
SwiGLU, RMSNorm, pre-norm, factorized `d_emb` 128 embeddings.

**The bet the project started on** was *weight sharing*: run one transformer layer
in a loop 12 times (ALBERT/Universal-Transformer style), storing 3M parameters but
computing as if it had 12 layers. The hope was that looping would buy most of a
deep model's quality at a fraction of the storage.

**What PR #7 established** is that this bet is half right, and the wrong half was
load-bearing. Untying the loop — `n_unique_layers` 1 → 12, `n_loops` 12 → 1,
*everything else identical* — is the `quark_3m` → `quark_22m` change. MEASURED:

| model | stored params | compute-equiv | word PPL | BLiMP | VRAM |
|---|---:|---:|---:|---:|---:|
| `quark_3m` (1×12, tied) | 2,868,352 | 20,643,840 | 115.16 | 57.05 | ~11.0 GB |
| `quark_22m` (12×1, untied) | 21,800,320 | 20,643,840 | **74.965** | **61.76** | ~11.3 GB |

The two run the **same compute graph** (identical FLOPs, ~identical VRAM), so the
20-point perplexity gap is bought purely by giving each loop iteration its *own*
weights. That is the essence in one sentence: **sharing saved storage, not
arithmetic — and VRAM, which is set by activations (compute), was the binding
constraint all along.** Stored weights were never what filled the 16 GB card.

![Fig 3 — sharing saves storage, not the compute that sets VRAM](report/fig3_stored_vs_compute.png)

## 2. Where quark actually stands (the honest competitive picture)

`quark_22m`'s 74.965 is still **2.0× GPT-2-124M's zero-shot 37.50** — but that
comparison is played on easy mode and the README/RESULTS.md §4.1 say so: GPT-2 is
48× larger *and* its 37.50 is out-of-domain (WebText excluded Wikipedia), so it
includes a domain-shift penalty quark does not pay. The word-PPL race is kept only
as a **secondary** metric; it is the wrong headline.

The right headline for a model this size is **grammatical competence (BLiMP)**, and
there the news is good: BLiMP has a cliff around 24–30M parameters (BabyLM data),
and quark scores **exactly where its parameter count predicts**. Its BLiMP is the
size class reporting in, not an anomaly to debug.

![Fig 2 — the BLiMP-vs-size cliff; quark lands on the curve](report/fig2_blimp_cliff.png)

## 3. The limiting factor

Two independent measurements point at the same wall, and it is not the one the
project originally feared (too few parameters):

1. **Token budget (Fig 4).** DERIVED from Chinchilla: `quark_22m` saw **6.5
   tokens/param** in its one epoch — **0.33× Chinchilla-optimal**. It is
   *undertrained, not undersized*. Four epochs would take it to ~26 tokens/param.
   This is why "more passes over the data" is the single largest expected win, and
   "more parameters" is not.
2. **VRAM (Fig 5).** The real 16 GB wall is the `seq²` attention matrix, i.e.
   activation memory, i.e. compute — not stored weights. That is why flash
   attention matters: it frees ~2.25 GB, which is what *funds* the batch-size and
   epoch experiments that the token-budget finding demands.

![Fig 4 — undertrained, not undersized: 0.33× Chinchilla](report/fig4_token_budget.png)

![Fig 5 — the 16 GB wall is the seq² attention matrix](report/fig5_vram.png)

**So the limiting factor is the token budget under the VRAM ceiling.** Every step
of the path below is chosen to push on exactly that, in the cheapest order.

## 4. The decided path forward

The order is the point — the exotic stuff is second-order until the cheap,
high-confidence wins are banked. This is NEXT.md §13, rendered as Fig 6:

1. **Fix the warmup unit** — ✅ *done in this PR* (§6). Cheap; everything downstream
   is an LR-adjacent sweep the old unit quietly corrupted.
2. **4 epochs + dropout 0.1, weight decay swept {0.1, 0.5, 1.0, 2.0}** — the
   headline. The one recommendation fitted at quark's own scale (Muennighoff 2023),
   and the largest expected win. *Everything below is noise until this runs.*
3. **Batch → ~64k tokens, then re-tune LR** — two scaling laws say quark is starved.
4. **Flash attention** — frees ~2.25 GB, no quality question; it funds 2–3.
5. **`d_emb` 128 → 256** — the cheapest untested lever (lifts the rank-128 softmax
   bottleneck).
6. Only then, one at a time: logit softcap, 14×352 depth, vocab 4096, AdEMAMix.

![Fig 6 — the gated roadmap; step 2 is the headline](report/fig6_roadmap.png)

The issue explicitly asked to "try different training methods and SOTA techniques
previously feared." The analysis's answer is deliberate: those techniques (softcap,
AdEMAMix, sliding-window, distillation, a neural tokenizer) are **step 6+**, not
step 1 — because until the token-budget wall (§3) is addressed they would measure
as noise. The harness encodes them as later, gated experiments rather than firing
them all at once.

## 5. The backend question: wgpu / vulkan / rocm

The issue asks which backend is fastest on the `gpu` runner. This is the **one
result that is wholly GPU-dependent**, and I want to be exact about its status:

- I **could not run it.** This branch is a fork PR; access to the upstream
  self-hosted runner is pull-only (403 on dispatch), and the environment I execute
  in has no GPU. I did not, and could not, execute any training run.
- Rather than invent a winner, I built the apparatus that produces the real answer
  and made the report render an honest placeholder for it (Fig 7):
  - `experiments/gpu/run_experiments.sh` times an **identical** short training run
    per backend and writes tokens/sec. Critically, it **compiles each backend
    outside the timed region**, so the first uncached backend is not charged ~12
    min of `rustc` against its throughput.
  - `.github/workflows/gpu-experiments.yml` runs it on `self-hosted` and
    uploads the rendered report + `results.json` as an artifact — so the outcome is
    reviewable without shelling into the runner.
- **How to get the number:** Actions → *GPU experiments* → Run workflow (defaults
  benchmark `wgpu vulkan`; add `rocm` to `bench-backends` if it links). Or locally:
  `BENCH_BACKENDS="wgpu vulkan rocm" ./experiments/gpu/run_experiments.sh`. The
  moment it runs, Fig 7 fills with measured bars and the placeholder disappears.

*Prior expectation, clearly labelled as expectation, not result:* on AMD, `burn`'s
Vulkan (SPIR-V) path is usually the safe first try and often edges `wgpu`; `rocm`
is untested here and may not even link. That is a hypothesis for the harness to
confirm or refute — it is **not** a finding.

## 6. The parameter sweep, and the one fix I landed

The issue asks to "quickly sweep non-obvious parameters." The harness encodes that
sweep (`experiments/gpu/gen_configs.py`), and every config is **derived from the
binary's own `quark train --dry-run`** — the AdamW constants and schedule shape are
never hand-written, so the experiments stay in lock-step with the code's defaults.
Fig 8 is the payoff panel: sweep word-PPL against the MEASURED `quark_22m` baseline
(74.965). It is a placeholder until the runner runs it.

![Fig 8 — the sweep vs the baseline (placeholder until run)](report/fig8_sweep.png)

**What I did change in code — NEXT.md §13 step 1, the warmup-unit fix.** This is
pure CPU logic, and the sweep is corrupted without it, so it was worth doing now:

- **The bug:** `burn` steps the LR scheduler once per *dataloader batch*, not per
  optimizer step. So `warmup_batches: 200` is a *different share* of every run —
  ~1.2% of the 1-epoch reference, but 0.3% of a 4-epoch sweep, and it shrinks again
  whenever `grad_accumulation` rises to hold a larger effective batch. The step-2
  sweep varies exactly those (epochs, batch split), so the old unit would have
  measured the warmup as much as the LR.
- **The fix:** `TrainConfig` now carries `warmup_ratio: Option<f64>`
  (`#[serde(default)]`, default `None`). When set, warmup is that *fraction of the
  whole run* — epoch- and grad-accumulation-invariant, because `total_batches`
  already folds both in. `None` reproduces the old absolute `warmup_batches`
  behaviour **exactly**, so the reference run is untouched and still reproduces the
  MEASURED 74.965. `effective_warmup_batches()` is the single source of truth read
  by both the scheduler and the run-length guard; `validate()` range-checks the
  ratio. Tests cover the fraction resolution, schedule equivalence to the matching
  absolute count, epoch/grad-accum invariance, the serde default, and validation.
- The sweep configs set `warmup_ratio = 0.01` (1% of the run, squarely in the
  standard 1–2% range and matching what the reference's 200 batches works out to),
  so every sweep run warms over the same share. The baseline keeps the absolute
  count, so it still reproduces the measured point.

## 7. Transparency: what ran, what did not, and how to finish it

To be unambiguous, because a graphical report can look more finished than it is:

- **No model was trained for this PR.** Every MEASURED number here and in the
  figures predates it (the three WikiText-103 runs in RESULTS.md and the PR-#7
  `quark_22m` point). I did not have GPU access.
- **What is new and runnable:** the `experiments/gpu/` harness, the
  `gpu-experiments.yml` workflow, the `warmup_ratio` code fix + tests, and the
  report generator with the two GPU panels wired to overlay real results the
  instant they exist.
- **To complete the empirical half** (the maintainer, on the machine with the GPU
  and the `_work/` data): Actions → *GPU experiments* → Run workflow, set
  `experiment-set` to `quick` first (reproduces 74.965, validates the pipeline in
  minutes), then `sweep`. It is time-boxed (`time-budget-hours`, default 6) and
  **logs every run it skips** rather than silently dropping it. Download the
  artifact; `docs/report/` regenerates with the placeholders replaced by measured
  bars and points.

## 8. Bottom line

The architecture is not the bottleneck and neither is the parameter count. PR #7
proved that the cheap structural win (untying the loop) is real and banked. The
next real gains are **tokens**, bought by **more epochs** and **freed VRAM**, in the
gated order of Fig 6 — with the exotic techniques kept for after that wall is
addressed, not before. The warmup unit that would have contaminated that sweep is
fixed. The only missing piece is GPU time, which I could not access; the harness
and workflow to spend it are in this PR and ready to run.
