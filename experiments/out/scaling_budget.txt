  3M capacity floor (Hoffmann)   : 4.241 nats  (PPL 69.5)
  GPT-2 124M on OWT, zero-shot   : 3.120 nats  (PPL 22.6)
  GPT-2 124M on OWT, finetuned   : 2.850 nats  (PPL 17.3)
  GAP vs zero-shot               : +1.121 nats  =>  3.07x worse PPL
  GAP vs finetuned               : +1.391 nats  =>  4.02x worse PPL

  ---- HONESTY BLOCK: how much does this argument actually prove? ----

  N=3e6 is 15x BELOW the smallest fitted model (44M), i.e. ~1.2
  decades outside a fit whose N-support spans only ~2.6 decades. Worse,
  it is self-contradictory under Chinchilla's own definitions: N counts
  embeddings, and with their 32k vocab a 3M-param TOTAL budget cannot
  even hold the embedding matrix at any sane d_model. The formula would
  be describing a model that is nearly all embedding and nothing like
  anything they fitted. So: DO NOT cite 4.24 nats as a prediction.

  What survives the caveats is the DIRECTION and the ORDER OF MAGNITUDE:
  both independent fits put the 3M floor ~1.1-1.4 nats above GPT-2's
  measured loss, and the floor is monotone in N under any fit with
  alpha > 0. The conclusion 'a 3M model cannot match a 124M model on
  the same distribution' does not depend on the disputed constants.

  The mechanism is what matters, and it is not subtle: the gap is at
  INFINITE data. So 'train on 15B tokens' cannot fix it, and neither
  can distillation -- a student cannot exceed its own capacity floor no
  matter how good the teacher is. Distillation changes WHICH function
  inside the student's hypothesis class you converge to; it does not
  enlarge the class.

3. Required 'effective parameter' multiplier to close the OWT gap
==============================================================================
  Hoffmann  vs zero-shot 3.12: need N=1.65e+07  ->    5.5x our budget
  Hoffmann  vs finetuned 2.85: need N=3.05e+07  ->   10.2x our budget
  Besiroglu vs zero-shot 3.12: need N=2.42e+07  ->    8.1x our budget

  Modern architecture (SwiGLU, RoPE, RMSNorm, better optimizer and LR
  schedule) is empirically worth roughly 1.2-2x in effective params.
  Every row above demands more than that, and the Besiroglu row demands
  vastly more. The verdict is robust to which fit you believe, which is
  exactly why we retarget rather than argue about constants.

4. The contestable target: WikiText-103 word-level PPL
==============================================================================
  GPT-2 124M scores 37.5 word-level PPL on WikiText-103 ZERO-SHOT
  (Radford et al. 2019, Table 3, "117M" row), and WebText explicitly excluded
  Wikipedia. So 37.50 is an OUT-OF-DOMAIN transfer number. We train ON the
  WikiText-103 train split (103,227,021 tokens): in-domain. That
  asymmetry -- not parameter efficiency -- is the edge, and it is worth a lot.

  This is not hand-waving. There is a published existence proof:

    DEQ-Transformer small (Bai et al. 2019, arXiv:1909.01377, Table 3)
      total params        : 138M
      NON-EMBEDDING params:  4.5M   <-- the actual transformer body
      WikiText-103 test   : 32.4 word-level PPL, in-domain
      (compare TrXL-small: 139M total -> 35.8)

  A 4.5M-parameter transformer BODY reaches 32.4, beating GPT-2 124M's 37.50.
  The 138M total is almost entirely vocabulary, which is the real lesson:
  word-level WikiText-103 is a vocabulary-STORAGE problem, not a modeling one.

  Why we cannot copy DEQ's setup: our 3M budget is TOTAL, embeddings
  included. A word-level output layer is arithmetically impossible.

  WikiText-103 vocab = 267,735 words. Tied embedding matrix alone:
     d_model   embedding params    vs 3.0M budget
          16          4,283,760              1.4x
          32          8,567,520              2.9x
         128         34,270,080             11.4x
         256         68,540,160             22.8x
         512        137,080,320             45.7x
    -> a 3.0M TOTAL budget caps word-level d_model at 11.2. Absurd.

  So we use a SUBWORD vocab (8192 BPE) and report word-level PPL by
  renormalizing: PPL_word = exp(total_NLL / n_words). This is EXACTLY
  the protocol GPT-2 used -- Radford et al: 'We evaluate the same
  quantity by computing the log-probability of a dataset according to a
  WebText LM and dividing by the number of canonical units.' GPT-2's own
  vocab (50257 BPE) is not word-level either. The metric is therefore
  tokenizer-independent and the comparison is legitimate.

  Per-token PPL across tokenizers is NOT legitimate: a smaller vocab
  mechanically lowers it (fewer choices per step, more steps per word).
  The harness must never report it as a cross-model comparison.

  ---- The one trap we cannot fully close ----
  GPT-2's 37.50 uses an 'invertible de-tokenizer' that undoes WikiText's
  <unk>/@-@/space-before-punctuation artifacts, worth 2.5-5 PPL by their
  own account. OpenAI never released it. So 37.50 is not exactly
  reproducible, and any number we compute without an equivalent
  de-tokenizer is not comparable to it. Mitigation: we re-evaluate the
  GPT-2 checkpoint OURSELVES through the same harness code path and
  report THAT as the baseline alongside the published 37.50.

  Softmax bottleneck (Yang et al. 2018, arXiv:1711.03953, Corollary 1):
  with tied+factorized embeddings the logit matrix factors through R^E,
  so rank(logits) <= E and the model provably cannot express the true
  distribution once rank(A) > E+1. Their Table 6 measures a d=400 softmax
  saturating at rank exactly 400 -- the bound is ACTIVE, not slack.
  research.txt's E=32 is a rank-32 cap on a real-language next-token
  distribution. That is the single worst decision in that document, and
  the reason quark-3m spends 1.05M of its 3.0M on embedding rank.

6. Training cost on a single 16GB GPU (wgpu backend)
==============================================================================
  Architecture: quark-3m (V=8192, E=128, H=384)
  Trainable params            : 2,868,352
  Compute-equivalent params   : 20,643,840  (7.2x the budget)

  Weight sharing means we PAY the FLOPs of a ~21M model while STORING a
  3M model. Budget the run against the former. This is the cost of the
  parameter trick, and it is not optional.

  D=3.0e+09 tokens: 5.41e+17 FLOPs  (matmul 3.7e+17 + attn 1.7e+17)
      4060Ti-16GB @10% MFU:    68.4 h  (  2.8 days)
      4080-16GB   @15% MFU:    21.5 h  (  0.9 days)
      optimistic  @30% MFU:    10.0 h  (  0.4 days)

  D=1.0e+10 tokens: 1.80e+18 FLOPs  (matmul 1.2e+18 + attn 5.7e+17)
      4060Ti-16GB @10% MFU:   227.9 h  (  9.5 days)
      4080-16GB   @15% MFU:    71.6 h  (  3.0 days)
      optimistic  @30% MFU:    33.4 h  (  1.4 days)

  D=1.5e+10 tokens: 2.71e+18 FLOPs  (matmul 1.9e+18 + attn 8.5e+17)
      4060Ti-16GB @10% MFU:   341.8 h  ( 14.2 days)
      4080-16GB   @15% MFU:   107.4 h  (  4.5 days)
      optimistic  @30% MFU:    50.1 h  (  2.1 days)

  WikiText-103 train is only 103M words, so an in-domain run is
  a few epochs = single-digit hours. The 3-15B token figures apply to
  the OpenWebText pretraining leg, which is days -- affordable but not
  free, and worth doing only if it demonstrably helps WT103.

7. Activation memory at seq=1024 (the real 16GB constraint)
==============================================================================
  Layer applications needing stored activations: 12
  Sharing does NOT reduce this. Every loop iteration's activations must
  be kept for backprop unless recomputed -- the shared layer sees
  different inputs each loop, so there is nothing to reuse.

   batch   act. GB   attn GB  total GB   fits 16GB?
       4      0.98      1.21      2.19          yes
       8      1.96      2.42      4.38          yes
      16      3.93      4.83      8.76          yes
      32      7.85      9.66     17.52           NO

  The materialized attention matrix dominates and scales as seq^2.
  Mitigations, in the order the harness applies them:
    1. gradient checkpointing per loop iteration (n_layer_apps -> 1)
    2. micro-batch + gradient accumulation to reach the target batch
    3. shorter seq (512) early in training, extend later
  Plan: micro-batch 8 @ seq 1024, accumulate to ~128-256 sequences.
